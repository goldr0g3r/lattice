## Summary

Implements the v0.3 hybrid query grammar locked by
[ADR-0018](docs/decisions/0018-search-query-grammar.md): a hand-rolled
lexer + recursive-descent parser produces a typed [`Query`](core/lattice-search/src/query.rs)
AST and surfaces parse errors with byte-range spans so the v0.3 search
modal ([#43](https://github.com/goldr0g3r/lattice/issues/43)) can
underline the offending region. Covers every operator the v0.3 epic
acceptance calls out — free text, `tag:` / `path:` / `created:` /
`updated:`, quoted phrase, prefix (`raft*`), fuzzy (`raft~` / `raft~2`),
boolean `OR`, negation, parens — and ships ~30 unit tests pinning the
AST shape.

Pure `lattice-search` + small `lattice-core` error-variant work; the
Tantivy execution layer (`Index::search(parsed)`), the `search_query`
Tauri command, and the modal UI come in v0.3 PR E
([#43](https://github.com/goldr0g3r/lattice/issues/43)). PR D exposes
[`lattice_search::Index::reader_searcher`](core/lattice-search/src/index.rs)
+ [`lattice_search::Index::tantivy`](core/lattice-search/src/index.rs)
(landed in [#63](https://github.com/goldr0g3r/lattice/pull/63))
as the execution bridge PR E will use.

- **AST** ([`core/lattice-search/src/query.rs`](core/lattice-search/src/query.rs))
  — `#[non_exhaustive]` for v0.9 plugin SDK extensibility:
  `All`, `Term`, `Phrase`, `Prefix`, `Fuzzy { term, distance }`,
  `Field { field, value }`, `Date { field, op, date }`, `And`, `Or`,
  `Not`. `DateOp` enum (`Gt`, `Ge`, `Lt`, `Le`, `Eq`) maps directly
  to a SQL `WHERE` clause the v0.3 PR E executor will compose against
  `notes.created` / `notes.updated`.
- **Lexer** — hand-rolled, handles whitespace, identifiers with
  internal hyphens (`bge-small` and `2026-01-01` lex as one token —
  this took two iterations to get right; the v0.3 PR E executor
  depends on date literals surviving the lexer as one token), quoted
  strings with `\"` / `\\` escapes, parens, colon, wildcard, tilde,
  minus, comparison operators (`<`, `<=`, `>`, `>=`, `=`), and the
  `OR` keyword — uppercase-only so users searching for the literal
  word "or" don't need to escape it.
- **Parser** — recursive-descent with precedence `OR < AND < NOT <
  ATOM`; flattens nested `And` / `Or` so `a b c` produces one
  `And(vec![a, b, c])`. `parse_not` recurses on itself so `--foo`
  collapses to `foo`. Refuses date operators on non-date fields with
  a clear "use `created` or `updated`" hint **before** trying to
  parse the value — better error UX than the date-parse error
  bubbling up first.
- **Field operators** recognised natively: `tag:foo`,
  `path:Engineering/`, `created:>2026-01-01`,
  `updated:<=2026-05-15`, with shorthand `created:2026-01-01` for
  implicit equality.
- **Defaults** locked here so the v0.3 PR E executor reproduces them
  without drift: `DEFAULT_FIELDS = ["title", "body", "tags"]`,
  `TITLE_BOOST = 3.0` (PR E uses `QueryParser::set_field_boost` to
  apply it), `DEFAULT_FUZZY_DISTANCE = 1`. Fuzzy distance caps at 2
  to match Tantivy's `Term::FuzzyTermQuery` limit.
- **Errors** — new
  [`SearchError::InvalidQuery { query, span, reason }`](core/lattice-search/src/error.rs)
  variant. The span is `std::ops::Range<usize>` so the modal can
  underline the byte slice with one substring call. `lattice-core`'s
  `From<SearchError>` impl special-cases `InvalidQuery` and lifts it
  to a new
  [`LatticeError::InvalidQuery { query, reason, span_start, span_end }`](core/lattice-core/src/error.rs)
  variant. The range is split into two `u32`s because `ts-rs` can't
  represent a `Range` natively without a custom type binding —
  cheaper to split. ts-rs export +
  [`error_snapshot__invalid_query_variant_shape.snap`](core/lattice-core/tests/snapshots/error_snapshot__invalid_query_variant_shape.snap)
  lock the IPC JSON shape:
  `{ "kind": "invalid_query", "details": { "query", "reason", "span_start", "span_end" } }`.
- **Empty / `*`** both parse to `Query::All` — matches every doc, so
  the modal can show the recent-notes list with the same code path.
- **`Cargo.toml`** — adds `chrono` to `lattice-search` (was implicit
  via `ts-rs` features only; the parser uses `NaiveDate` directly).

## Test plan

- [x] **30 new unit tests in `query::tests`** covering: empty matches
      all, `*` matches all, bareword, implicit AND, three-word AND
      flatten, `OR` keyword vs lowercase `or`, precedence (`OR` below
      AND), parens override, single + double negation,
      dashed-identifier-is-one-term (`bge-small`), quoted phrase
      (with `\"` escape), prefix, fuzzy default + explicit distance +
      cap at 2, field-scoped term + phrase + prefix, date `>`,
      `<=`, implicit `=`, negated field query (`-tag:draft`), a
      complex combined query, every error path (unterminated quote,
      unclosed paren, unbalanced close, bad date with span, date op
      on non-date field, empty field value), whitespace tolerance,
      nested parens, and a constant-drift guard against ADR-0018's
      `DEFAULT_FIELDS` + `TITLE_BOOST`.

      (was 11 / 11 after [#63](https://github.com/goldr0g3r/lattice/pull/63)).

      (was 75 / 75 after [#64](https://github.com/goldr0g3r/lattice/pull/64);
      `error_snapshot.rs` grows `invalid_query_variant_shape`).
- [x] `cargo fmt --all --check` — clean.

## Out of scope (deferred to follow-up PRs)

- **`Index::search(query: &Query, limit: usize)`** that lowers the
  AST to a `tantivy::query::Query` and returns ranked
  `SearchHit { id, path, title, snippet, score }` results — ships in
  [#43](https://github.com/goldr0g3r/lattice/issues/43)'s PR.
- **SQLite-side date filtering** that joins `Query::Date` ranges
  against `notes.created` / `notes.updated` — ships in
  [#43](https://github.com/goldr0g3r/lattice/issues/43)'s PR.
- **`search_query` Tauri command + the React search modal** — ship
  in [#43](https://github.com/goldr0g3r/lattice/issues/43)'s PR
  (waiting on the visual-polish PR to merge before any
  `apps/desktop/src/shell/*.tsx` edit).
- **Inline `^boost` syntax** (e.g., `raft^2`) — explicitly deferred
  by [ADR-0018](docs/decisions/0018-search-query-grammar.md);
  v0.3 boosting comes from the locked default-field list only.
- **Numeric-field range queries** (e.g., `size:>1024`) — deferred to
  v0.7 (typed blocks bring numeric fields).
